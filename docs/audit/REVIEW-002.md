| C1 | `src/schemas/customer.js:5-8` ↔ `src/lib/db/customers.js:7` | **entity_type enum mismatch — CustomerForm completely broken.** The form schema (`customerSchema`) uses display values: `'Sdn Bhd'`, `'Enterprise'`, `'Individual'`, `'Cooperative'`, `'Gov Agency'`, `'Other'`. When the form submits, `createCustomer()` calls `CustomerInsertSchema.parse(data)` which rejects display values, throwing a ZodError. Same issue in `updateCustomer()`. The CustomerForm is completely non-functional for the entity_type field — every create/update will fail. | Align `customerSchema.entity_type` in `src/schemas/customer.js` to use the DB snake_case values (`'sdn_bhd'`, `'enterprise'`, etc.) and translate to display labels only in the `<select>` options (use `value={enumValue}` + display text separately). Alternatively, add a mapper in `createCustomer`/`updateCustomer` to translate display → DB values before validation. Fix the select options in `src/pages/customers/CustomerForm.jsx:90` to use DB values. |
| C2 | `src/pages/vehicles/DocumentCabinet.jsx:7` ↔ `supabase/migrations/004_vehicle_documents.sql:6` | **doc_type value `sld_cert` does not match DB CHECK constraint.** DocumentCabinet uses `sld_cert` as the doc_type key (line 7 of DocumentCabinet.jsx). The DB CHECK constraint (migration 004, line 6) only allows `sld`. Any upload attempt for an SLD certificate will fail with a CHECK constraint violation at the DB level. The Dashboard `DOC_TYPE_LABELS` map (`src/pages/Dashboard.jsx:11`) also uses `sld_cert` as its key. | Change `sld_cert` → `sld` in `src/pages/vehicles/DocumentCabinet.jsx:7` and `src/pages/Dashboard.jsx:11`. | C3 | `src/lib/db/vehicles.js:16` | **VehicleInsertSchema status enum out of sync with DB.** The schema allows `['active', 'sold', 'scrapped']` and declares `'sold'` which does not exist in the DB CHECK constraint (`active`, `in_shop`, `decommissioned`, `scrapped` from migration 003:31). It also omits `'in_shop'` and `'decommissioned'` which ARE valid in the DB and ARE used by the form schema (`src/schemas/vehicle.js:37`). Creating a vehicle with `status: 'in_shop'` (which passes the form schema) will fail when `VehicleInsertSchema.parse()` rejects it. | Update `src/lib/db/vehicles.js:16` to match the DB constraint: `z.enum(['active', 'in_shop', 'decommissioned', 'scrapped'])`. Remove the orphaned `'sold'` value. |
| C4 | `src/pages/vehicles/DocumentCabinet.jsx:71-80` | **DocumentCabinet bypasses the Storage atomicity fix.** Commit 0881c60 added rollback logic to `src/lib/db/documents.js`'s `uploadDocument()` that deletes the Storage file if the DB insert fails. But DocumentCabinet calls `storageUpload()` (from `storage.js`, just uploads to Storage) and then `upsertDocument()` (from `documents.js`, just inserts DB record) as two separate calls. The rollback in `uploadDocument()` is never triggered. If the DB insert fails after a successful Storage upload, the file is orphaned. | Either (a) refactor DocumentCabinet to use the combined `uploadDocument` from `documents.js` instead of separate `storageUpload` + `upsertDocument` calls, or (b) wrap the two calls in DocumentCabinet with a try/catch that removes the Storage file if `upsertDocument` fails (mirroring the rollback in `documents.js`). |

## High (should fix before merge)

| # | File:line | Issue | Fix |
|---|---|---|---|
| H1 | `src/pages/customers/CustomerForm.jsx:61` | **Stale `id` in `onSuccess` cache invalidation.** When `isEdit` is false (create mode), `id` from `useParams()` is `undefined`. `queryClient.invalidateQueries({ queryKey: ['customer', id] })` becomes `['customer', undefined]`. While not harmful (just invalidates a non-existent key), it becomes a real problem if the queryCache uses `undefined` as a sentinel — plus it misleads maintainers. | Replace with a conditional: `if (isEdit) queryClient.invalidateQueries({ queryKey: ['customer', id] })`. Same pattern in `src/pages/vehicles/VehicleForm.jsx:79`. |
| H2 | `src/pages/Dashboard.jsx:94-110` | **Dashboard queries have no error handling.** All three `useQuery` calls (lines 94, 100, 106) omit `throwOnError` and the render code never checks `isError`. If `getExpiryAlerts`, `getFleetAgeStats`, or `getQuickStats` fail (network error, RLS violation, view missing), the UI silently shows `—` values with no user-facing error indicator. The queries will retry once (from `main.jsx:31` default), then stay in error state silently. | Add error handling to each query or at a minimum add a consolidated error banner. Example: add `const queryErrors = [alertsError, ageError, statsError].filter(Boolean)` and render an error banner if any exist. |
| H3 | `src/pages/vehicles/VehicleList.jsx:33-35` | **`replacementOnly` filter not synced with URL changes after initial mount.** The state is initialized from `searchParams` on first render: `useState(searchParams.get('age') === 'replacement')`. But URL changes while on the page (browser back/forward, programmatic navigation) will NOT update the filter. The Dashboard link `/vehicles?age=replacement` works on navigation but not on history traversal. | Use a `useEffect` to sync `replacementOnly` with `searchParams`, or derive the value from `useSearchParams` directly instead of storing in local state. |
| H4 | `src/pages/customers/CustomerForm.jsx:52-54` | **`reset(existing)` includes nested relational data.** `getCustomerById()` returns the customer with a nested `vehicles` array (from the Supabase join). Calling `reset(existing)` in the useEffect floods the form with fields not in the schema (`vehicles`, `customer_id`, `created_at`, `updated_at`). While react-hook-form strips unknown fields on submit, the reset itself sets them as form values, which can cause unexpected behavior with `dirtyFields` or if the schema evolves. | Destructure only the fields needed: `if (existing) { const { vehicles, customer_id, created_at, updated_at, ...rest } = existing; reset(rest); }`. Same pattern in `src/pages/vehicles/VehicleForm.jsx:70`. |
